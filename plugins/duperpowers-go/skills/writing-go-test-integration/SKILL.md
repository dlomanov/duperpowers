# Writing Go Integration-Test

Convention for writing Go integration tests: white-box, repository-only, isolated by transaction rollback. Real DB, no mock for the storage itself; external deps (timer, ID generator, queue) are mocked.

> **Sibling skill:** for non-repo unit tests (usecases, gRPC handlers, kafka, adapters), use `writing-go-test-unit`.
> **Maintenance:** rules R1, R3-R12, R15, R18, R21, R22 are duplicated with `writing-go-test-unit`. Universal-rule changes must be applied to both skills.

## The Iron Law

```
NO NEW TEST WITHOUT READING THE PACKAGE'S EXISTING TESTS FIRST
```

The whole package shares one `mockList`, one `makeSUT`/`makeMocks`, one `before` shape and one assertion style (R3-R7). If you haven't read them, you can't reuse them.

**Violating the letter of the rules is violating the spirit of the rules.**

## Reference implementation

A complete, compiling integration-test example accompanies this skill - read it alongside the rules:

`examples/repo/` (`repo.go`, `create.go`, `list.go`, `update.go`, `repo_test.go`, `create_test.go`). Demonstrates all rules in this skill in a real repository.

Each rule below carries a `[example: <path>:<lines>]` reference into this tree.

## 1. When and how we write tests

We treat tests as white-box. We know the SUT's implementation and reach maximum coverage with a minimum of cases. We do not write many cases that probe the SUT from every angle.

Goals:
- increase coverage
- check functionality (code compiles and runs as expected against a real DB)
- pin down behavior (characteristic tests)

Integration tests cover ONLY repositories. Everything else (gRPC handlers, kafka, usecases, adapters) is covered by unit tests.

**R1. White-box.** Tests live in the same package as the implementation under test (no `_test` package suffix).  [example: examples/repo/repo_test.go:3]

**R2. Integration tests are for repositories only.** Unit tests cover everything else - see `writing-go-test-unit`. This skill is for repository tests.

## 2. Infrastructure

**R3. Reuse existing infrastructure.** Before writing a new test, ALWAYS read every `*_test.go` in the package. Identify the package's `mockList`, `makeSUT` / `makeMocks`, `before` patterns, and assertion style - you MUST reuse it.

**R4. Main test file: location and layout.**
Test infrastructure (`mockList`, `makeSUT`, `makeMocks`, `TestMain`, the shared `testDB` and `errRollback`) MUST live in the package's main test file:
- repository → `repo_test.go`

Section order inside that file MUST be:
1. `const (...)`
2. `var (...)` - including `errRollback`, `testDB`, package-shared fixtures
3. `type (...)` / `type mockList struct`
4. `TestMain` - opens the real DB connection, runs `m.Run()`, closes
5. `makeSUT()` / `makeMocks()`
6. tests (headline `TestRepo` first, per-method tests after)

[example: examples/repo/repo_test.go:1-51]

**R5. Mock generation: `make mock` only.**
Mocks MUST BE ALWAYS generated by `make mock` (under the hood - `mockery`).
Never write mocks by hand.
Every project's `Makefile` defines a `mock` target.

Repository tests usually mock external dependencies (timer, ID generator, queue) even though the DB itself is real.

[example: examples/repo/repo_test.go:17,34]

**R6. One `mockList` per test package.**
Each test package MUST ALWAYS have exactly one `mockList` struct holding **pointers** to all generated mocks the package uses.
Mocks are constructed in `makeSUT` / `makeMocks`, never elsewhere.  [example: examples/repo/repo_test.go:28-30]

**R7. SUT factories: `makeSUT` or `makeMocks`.**
When the package has one SUT, write `makeSUT(t)` returning `(SUT, mockList)`.
When the package has more than one SUT, write `makeMocks(t)` returning `mockList` and inline the SUT construction in each test.
NEVER place `t.Helper()` inside `makeSUT` / `makeMocks` - they construct things, they are not helpers.

In integration tests, `makeSUT` ALSO sets up package-shared mock expectations (`.Maybe()`) for fixtures every test reuses (e.g. `timer.NowUTC()`).

[example: examples/repo/repo_test.go:32-39]

**R8. Test naming.**
Test names MUST ALWAYS use one of next patterns:
- `Test{Type}_{PublicMethod}` / `Test{Type}_{privateMethod}` for methods
- `Test_{PublicFunc}` / `Test_{privateFunc}` for package-level functions
- `Test{Type}` for the headline integration test of a repository - one per repo, exercises every method together (see **R27**). Example: `TestRepo`.

Test package MUST be same as implementation package => no `_test` suffix (because of whitebox).

[example: examples/repo/create_test.go:14, examples/repo/repo_test.go:53]

**R9. Parallelism + context.** Always parallel:
- `t.Parallel()` at the start of the outer test
- `t.Parallel()` at the start of each `t.Run` body, followed by a blank line
- Use `t.Context()` instead of `context.Background()`

```go
func TestRepo_Create(t *testing.T) {
    t.Parallel()

    sut, _ := makeSUT(t)
    // ...
    for _, tt := range tests {
        t.Run(tt.name, func(t *testing.T) {
            t.Parallel()

            r := require.New(t)
            err := sut.db.WithTx(t.Context(), func(ctx context.Context) error {
                tt.action(ctx, r)

                return errRollback
            })
            r.ErrorIs(err, errRollback)
        })
    }
}
```

[example: examples/repo/create_test.go:14-15,37-41,46]

**R10. Mock chain formatting.** One method per line; break after `EXPECT().`

```go
// BAD
m.timer.EXPECT().NowUTC().
    Return(testTime).Maybe()

// GOOD
m.timer.EXPECT().
    NowUTC().
    Return(testTime).
    Maybe()
```

(Integration mock setup is sparse compared to unit - usually a single `.Maybe()` chain inside `makeSUT`. The rule still applies wherever an `EXPECT().` chain appears.)

[example: examples/repo/repo_test.go:36]

**R11. Assertion library: `testify` defaults.**
Use package-level `assert.*` / `require.*` by default.
Allocate `r := require.New(t)` / `a := assert.New(t)` ONLY when the test makes more than 3 assertion calls.
In integration tests, default to `testify/require` - switch to `assert` only when failure should NOT abort (the headline `TestRepo` uses `assert` so the long happy-path keeps reporting after the first divergence; per-method tables use `require` to fail fast - see **R28** for the WHY).

[example: examples/repo/repo_test.go:89, examples/repo/create_test.go:40]

**R12. Error assertions.**
ALWAYS use `assert.ErrorIs` / `require.ErrorIs` (per **R11**) - handles nil and non-nil uniformly.
ALWAYS use `assert.AnError` for "any error" - never `var errTest = errors.New(...)`. The package-level `errRollback` (used to abort the transaction) is the ONE allowed exception - it is a control-flow signal, not a test fixture.

```go
// BAD
if tt.wantErr != nil { require.ErrorIs(t, err, tt.wantErr) } else { require.NoError(t, err) }

// GOOD - one line, both cases
require.ErrorIs(t, err, tt.wantErr)
```

```go
// BAD
var errTest = errors.New("test error")
wantErr: errTest,

// GOOD
wantErr: assert.AnError,         // any error
wantErr: domain.ErrNotFound,     // specific production sentinel
```

| Rationalization | Reality |
|---|---|
| "Just this once I'll define a custom `errTest = errors.New(...)` for clarity" | No - `assert.AnError` is the canonical sentinel. Custom test errors fragment the codebase and break grep-ability across packages. |
| "Comparing error strings is fine for a one-off test" | No - string compare breaks under wrapping (`fmt.Errorf("ctx: %w", err)`). Use `ErrorIs`. |
| "I need a second sentinel besides `errRollback`" | No - `errRollback` is the only allowed package-local sentinel because it controls transaction rollback. Any other "error" sentinel = `assert.AnError`. |

[example: examples/repo/repo_test.go:23,95, examples/repo/create_test.go:46]

## 3. Integration tables and bodies

The integration table shape is **NOT** the same as the unit shape - integration uses `name + action` (the action contains arrange + act + assert inside the rollback closure), not `args / before / want / wantErr`.

**R13. Table tests: when and shape (integration).**
ALWAYS use a table even for a single case - new cases will be added as the repo grows, and starting from a table makes extension trivial (see **R27**).
Stick to one template per file. Field order: `name`, `action`.
Always multi-line struct literals - every field on its own line, no compact one-liners.
Case names use spaces: `"execx error"`, `"missing FK"`. The happy-path case is `"success"`, never `"happy_path"`.

```go
// example template
tests := []struct {
    name   string
    action func(ctx context.Context, r *require.Assertions)  // or *assert.Assertions for TestRepo
}{
    // example: examples/repo/create_test.go:19-35
}
```

[example: examples/repo/create_test.go:19-35, examples/repo/repo_test.go:58-84]

**R14. `action` field - the whole AAA inside the rollback closure.**
The `action` closure runs INSIDE `sut.db.WithTx(ctx, func(ctx) error { tt.action(ctx, r); return errRollback })`. Signature: `func(ctx context.Context, r *require.Assertions)` for per-method tables, `func(ctx context.Context, a *assert.Assertions)` for the headline `TestRepo`.
The closure does Arrange (set up DB state) + Act (call the SUT) + Assert (check results). All three live inside one closure because all DB writes must roll back together.

[example: examples/repo/create_test.go:21-34,41-46, examples/repo/repo_test.go:60-83,90-95]

**R15. Case comments - explain WHY.** Above each case, a short comment on the behavior or invariant the case guards. No comments that restate the case name.
NEVER RESTORE COMMENTS THE USER MANUALLY DELETED.

```go
// BAD - restates name
// execx error
{
    name: "execx error",
    action: ...,
},

// GOOD - explains WHY
// canceled context must surface as an error from execx
{
    name: "execx error",
    action: ...,
},
```

[example: examples/repo/create_test.go:23,62]

**R17. AAA inside the action closure.** Inside the `action`, separate Arrange / Act / Assert with one blank line. No section markers.

```go
// GOOD - blank lines separate AAA inside the action
action: func(ctx context.Context, r *require.Assertions) {
    ctx, cancel := context.WithCancel(ctx)
    cancel()

    err := sut.Create(ctx, models.Order{})

    r.Error(err)
},
```

[example: examples/repo/create_test.go:26-33]

**R18. Case order - simple → complex → success LAST.** Reader sees "what can go wrong" before "what goes right". For per-method tables this means error cases first, success last. The headline `TestRepo` is a single happy `success` case, so this rule applies only to per-method tables (R27).

[example: examples/repo/create_test.go:23-34]

**R19. One case per scenario, not per field.** A test case encodes an integration scenario - canceled context, missing FK, conflict, success. A new field added to the model **extends the existing `success` case** in the headline `TestRepo`: update the `makeOrder()` builder or the inline literal. Add a new per-method case ONLY when the field unlocks a new failure path (new constraint, new branch).

| Rationalization | Reality |
|---|---|
| "A new field is clearer as a separate case" | A case encodes a scenario, not a field. Field propagation lives in the shared builder; duplicate scenarios add noise only. |

**R21. Test data - `gofakeit` vs hardcoded.**
YOU MUST USE `gofakeit` for test-values by default.
YOU MUST USE specific values ONLY if that value has meaning for the test-case.
The package-shared `testTime` is the canonical example: hardcoded once because every row needs a stable, comparable timestamp; everything else (UUIDs, totals, statuses) is `gofakeit`.

[example: examples/repo/repo_test.go:24,100-108]

**R22. Shared values - `var(...)` block at the top of the main test file.**
Package-shared values - `errRollback`, `testTime`, `testDB`, fixture builders - live in a `var(...)` block at the top of `repo_test.go`. Per-test fixtures (a fresh `models.Order` per case) are built via a helper like `makeOrder()`.

```go
var (
    errRollback = errors.New("rollback error")
    testTime    = gofakeit.Date().UTC().Truncate(time.Microsecond)
    testDB      *postgres.DB
)
```

[example: examples/repo/repo_test.go:22-26,100-109]

## 4. Integration-specific

**R25. Transaction-rollback isolation.** Every integration test runs inside `sut.db.WithTx(ctx, func(ctx) error { ...; return errRollback })`. Returning `errRollback` aborts the transaction, leaving the DB clean for the next test - this is what makes the package-shared `testDB` safe for parallel tests. The outer `WithTx` call returns `errRollback`, which the test asserts with `ErrorIs(err, errRollback)` to confirm the rollback fired.  [example: examples/repo/repo_test.go:90-97, examples/repo/create_test.go:41-47]

**R26. `go:build integration` tag.** Add the tag at the top of every integration `*_test.go` file when the project uses a build-tag split.  [example: examples/repo/repo_test.go:1, examples/repo/create_test.go:1]

**R27. Repository test shape: one happy `TestRepo` + per-method tables for the rest.**
Write one long happy-path `TestRepo` that exercises every method on the repository in sequence (Create + Update + List + ...). Use `assert` inside it (per **R11**) so a single mismatch does not abort the whole walkthrough.
Cover negative cases the happy path does not reach with per-method tests (`TestRepo_Create`, `TestRepo_Update`, ...). Each per-method test is a table even when there's only a single case - new cases will be added as the repo grows, and starting from a table makes extension trivial.  [example: examples/repo/repo_test.go:53-98, examples/repo/create_test.go:14-49]

**R28. Default to `testify/require` in per-method integration tables; `assert` in the headline `TestRepo`.**
Per-method tables use `require` to fail fast - a half-failed integration test typically leaves shared state in a bad shape (even with rollback, partial progress through `action` is not informative). The headline `TestRepo` uses `assert` so the long happy-path reports every divergence in one run.  [example: examples/repo/create_test.go:9,40,46, examples/repo/repo_test.go:14,89,95]

## 5. Golden Rules

Most-violated rules. If the code doesn't follow them, that's the first thing to fix.
- **R3** Read existing tests first - the package's `mockList`, `makeSUT`, `testDB`, `errRollback` are already there
- **R12** `ErrorIs` + `assert.AnError` - never if/else, never custom `errTest` (only `errRollback` is allowed)
- **R13** Always multi-line struct literals - never compact one-liners; always a table even for a single case
- **R14** `action` is AAA inside the rollback closure - not arranged outside
- **R25** Transaction-rollback isolation: `WithTx` + `return errRollback`, asserted with `ErrorIs`
- **R27** TestRepo (assert, long happy) + per-method tables (require, negatives)
- **R28** `require` in per-method tables, `assert` in headline `TestRepo`
